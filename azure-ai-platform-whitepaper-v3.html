00 — Executive Summary

What We Built
& the Decisions Behind It

This is not a theoretical reference architecture. Every layer described in this paper has been built and is operational. More importantly, every non-standard choice was made deliberately — balancing speed of delivery, operational pragmatism, and long-term governance.

The Cloud-Agnostic AI Platform (CAAP) on Azure is designed around a foundational principle: AI infrastructure should be governed, reproducible, and portable. On Azure, this is realised through four layers — a policy-governed landing zone, a credential-free CI/CD pipeline, a fully provisioned AI Foundry infrastructure, and a semi-automated MLOps pipeline built on Azure ML.

Where industry convention might demand full automation, we made deliberate, informed choices to introduce human governance at the right points — specifically around model promotion to production. This reflects mature ML engineering thinking, not a limitation of the platform. The infrastructure to go further is already in place.

01 — Bootstrap & Landing Zone

Foundation:
Azure Landing Zone at Enterprise Scale

The landing zone is the non-negotiable starting point. Getting it right means every workload, every team, and every AI service inherits security, governance, and network segmentation without any per-team configuration.

Delivered — ALZ Accelerator · Azure Verified Modules

1.1 Why Azure Landing Zone Accelerator with AVM

Rather than engineering a bespoke Management Group hierarchy and policy framework from scratch, we adopted the Azure Landing Zone Accelerator implemented entirely through Azure Verified Modules (AVM) — Microsoft's officially maintained, semantically versioned Terraform module library. This was a deliberate architectural choice that delivered three compounding benefits.

First, it eliminated months of foundational design work while giving us a governance framework battle-tested across thousands of enterprise deployments. Second, AVM modules receive upstream security improvements and new Azure feature support — our landing zone improves without us maintaining the underlying module code. Third, the consistent AVM module interface design means our Azure implementation documents cleanly alongside equivalent AWS and GCP implementations in future tracks.

1.2 Multi-Region Hub-Spoke Topology

The network follows a hub-spoke model deployed across two Azure regions. A dedicated Connectivity subscription hosts the hub VNet in each region containing Azure Firewall, Azure DDoS Protection, Azure DNS Private Resolver, VPN/ExpressRoute Gateways, and centralised Private DNS Zones. All spoke VNets peer to the regional hub, with all egress routing enforced through Azure Firewall policies.

This topology is fundamental for the AI Platform spoke: Azure AI Foundry, APIM, Azure ML, and all backing services are accessed exclusively via Private Endpoints within the spoke network. DNS resolution for all private endpoints flows through the hub's centralised Private DNS Resolver — ensuring consistent name resolution across all spokes and future on-premises connectivity via ExpressRoute.

Diagram 01 — Multi-Region Landing Zone & Hub-Spoke Network AVM · ALZ Accelerator

1.3 Key Best Practice Decisions

Private DNS centralisation in the hub is enforced via Terraform output references — the AI Platform spoke module receives hub DNS Zone IDs as inputs and all Private Endpoint DNS registrations target hub zones exclusively. This prevents the common split-brain DNS problem where spokes create their own zones and break cross-spoke resolution.

Azure Policy deny effects are inherited from the ALZ initiative and block manual resource creation in production subscriptions. Combined with Azure Locks on critical resources (Terraform state storage, Key Vaults), this makes the infrastructure resistant to configuration drift without relying on team discipline.

02 — CI/CD Pipeline

Automation:
Credential-Free Infrastructure Delivery

Every infrastructure change is version-controlled, reviewed, and auditable. No manual Azure portal changes. No long-lived credentials anywhere in the system. This is a non-negotiable baseline for a platform that manages AI infrastructure at scale.

Delivered — GitHub Actions + OIDC Workload Identity Federation

2.1 OIDC Workload Identity — Why This Matters

The most consequential security decision in the CI/CD design is authentication. GitHub Actions pipelines authenticate to Azure using OIDC Workload Identity Federation, eliminating client secrets, certificates, and passwords entirely. This is the current industry best practice for IaC pipeline authentication — and it is implemented here from day one rather than retrofitted.

The trust relationship is established at the Azure AD Application level. GitHub's OIDC issuer is federated to an Azure service principal scoped to the minimum required permissions. Each pipeline run receives a short-lived token valid only for that run's duration. Exposed pipeline logs cannot leak reusable credentials because none exist.

.github/workflows/platform.yml — auth blockGitHub Actions
# No secrets — OIDC token issued at runtime per workflow run
permissions:
  id-token: write    # Required to request OIDC token
  contents: read

jobs:
  terraform:
    environment: production   # Approval gate enforced at env level
    steps:
      - uses: azure/login@v2
        with:
          client-id:       ${{ vars.AZURE_CLIENT_ID }}
          tenant-id:       ${{ vars.AZURE_TENANT_ID }}
          subscription-id: ${{ vars.AZURE_SUBSCRIPTION_ID }}
          # ↑ Non-secret variable IDs — not credentials

      - run: terraform init -backend-config="use_oidc=true"
      - run: terraform apply -auto-approve tfplan.out
        # ↑ Only runs against pre-approved, hash-verified plan artifact

Terraform Plan + Artifact

Plan runs against the target environment. The plan artifact is saved to Azure Blob Storage (same account as Terraform state, same RBAC boundary) and its SHA-256 hash is recorded. This is the foundation of plan integrity — the hash is verified before apply, preventing any modification between review and execution.

03
human gate
Approval Gate — GitHub Environments

Production deployments require explicit approval from a Platform Engineer via GitHub Environments. The approval record is permanently stored in GitHub's audit log. Non-production environments auto-apply after a clean plan — a deliberate choice to keep dev iteration fast while protecting production.

04
automated
Apply & Smoke Test

The hash-verified plan artifact is applied. Post-apply smoke tests verify key resources are in a healthy provisioning state. Failures trigger an automatic rollback notification with the full state diff. Terraform state is stored in Azure Blob Storage and accessed exclusively via OIDC — no human has standing access to production state files.

Architectural Decision — Confirmed Best Practice
Plan artifact integrity verification

Saving the Terraform plan as a verified artifact and checking its hash before apply closes a subtle but real attack vector: the plan-to-apply drift attack, where infrastructure changes between the reviewed plan and the execution. This pattern is recommended in the HashiCorp Terraform security model and is implemented here from the start.

03 — AI Foundry & GenAI Infrastructure

AI Platform:
Full Infrastructure Substrate via AVM

The AVM ai-ml pattern module provisions the complete governed infrastructure foundation for all AI and GenAI workloads. This is not a partial deployment — it is the full dependency stack, fully private, fully policy-compliant, and ready for application teams to build on top of.

Delivered — AVM ai-ml Pattern Module

3.1 What the AVM ai-ml Pattern Module Deploys

The module follows a clear and deliberate philosophy: provision the governed platform substrate, not the applications. Every AI service dependency, every Private Endpoint, every network subnet, every identity binding — all provisioned as code through the AVM module. Application workloads are brought by teams on top of this foundation, through the same CI/CD pipeline.

This separation is architecturally significant. It means the platform team can upgrade, patch, and govern the AI infrastructure independently of application teams — and application teams can deploy and iterate within well-defined boundaries without needing to understand or modify the underlying platform.

Architectural Decision — Platform Boundary
Container App Environment provisioned empty — by design

The AVM module provisions the Container App Environment with all networking, identity, and governance configuration in place but with no running applications. It eliminates coupling between platform upgrades and application deployments, and enables multiple teams to independently deploy into the same environment under the same security posture.

04 — ML Workspace & MLOps Pipeline

Intelligent Operations:
Event-Driven MLOps with Governed Promotion

The MLOps layer is where infrastructure meets intelligence. Built on a custom-provisioned Azure ML Workspace, it delivers a semi-automated pipeline that handles routine operational decisions autonomously while keeping human engineers in control of production promotion — a balance that reflects mature, production-grade ML thinking.

Delivered — Custom Terraform · Azure ML Pipelines · Event Grid · Function App

4.1 ML Workspace — Custom Terraform Provisioning

The Azure ML Workspace is provisioned through a custom Terraform codebase, separate from the AVM ai-ml pattern module. This reflects a deliberate engineering choice: the ML Workspace has configuration requirements specific to ML workloads that warranted a purpose-built module rather than a generic platform pattern.

The workspace is configured in Allow Internet Outbound mode — a considered decision that balances two real constraints. ML compute needs outbound internet access to pull package dependencies, Docker base images, and pre-trained model weights from public registries during training jobs. Simultaneously, all data-plane resources (Storage, Key Vault, Container Registry, and the workspace itself) remain entirely private with dedicated Private Endpoints. This configuration gives ML engineers a practical developer experience without compromising data security.

Architectural Decision — Allow Internet Outbound Mode
Why not fully private compute?

A fully private ML workspace (no internet outbound) requires mirroring all package indices, Docker registries, and model hubs into private Azure resources — a significant operational overhead that adds friction to every new package or model version adoption. Allow Internet Outbound mode is Microsoft's recommended configuration for ML workloads that need package flexibility while keeping all data private. Data never leaves the private network; only compute dependency downloads use the outbound path. The platform is fully capable of enforcing a private-only configuration if compliance requirements demand it — the infrastructure exists to support both modes.

4.2 One-Time Setup — Intentional Onboarding, Not a Gap

Standing up a new ML environment involves a set of one-time setup steps performed by an ML engineer using the documented runbook: registering data assets in the workspace, creating and registering ML environments (Docker/Conda specifications), creating the training pipeline from the existing codebase, and wiring up the retraining loop components — the batch endpoint, Function App subscription, and Event Grid topic connection.

These steps are intentionally performed once per environment rather than fully automated. This design reflects a considered trade-off: ML environments carry context-specific configuration (dataset schemas, model architecture choices, evaluation thresholds) that benefits from engineer review at setup time. The runbook is fully documented and repeatable. The infrastructure to fully automate this onboarding exists and can be implemented as a Phase 2 enhancement when the pattern stabilises across teams.

4.3 The Semi-Automated MLOps Lifecycle

Once an environment is set up, the operational pipeline runs with minimal human involvement for routine retraining cycles. The architecture strikes a deliberate balance: automated where repetition adds no value (detecting drift, triggering retraining), and human-gated where judgement adds value (deciding whether a retrained model should go to production).

01
One-time per environment
Environment Setup — Documented Onboarding

An ML engineer registers data assets, creates ML environments (Docker/Conda), authors the training pipeline from the existing codebase, and wires the retraining loop: creates the batch endpoint, deploys the pipeline component, configures the Event Grid topic, and registers the Function App subscription. This is a deliberate, reviewed setup step — not a candidate for full automation until the pattern stabilises across teams. The runbook is fully documented and repeatable in under a day.

02
automated
Experiment & Training Pipeline

Training pipeline jobs run on Azure ML compute clusters. All runs are tracked with full lineage — input dataset version, hyperparameters, environment hash, and output artefacts. An evaluation pipeline automatically compares the challenger model against the current champion on a version-locked held-out dataset. Models that pass the quality threshold are automatically registered in the Azure ML Model Registry with a Staging tag and complete metadata.

03
automated
Continuous Monitoring — Data Drift & Model Drift

Azure ML's built-in monitoring runs continuously against the production endpoint, tracking input feature distribution (data drift) and prediction quality (model drift) against the training baseline. This is fully automated — no engineer needs to check dashboards to detect that retraining is needed. When drift exceeds the configured threshold, a signal is published to an Azure Event Grid topic.

04
automated
Event-Driven Retraining — Function App + Batch Endpoint

A custom Azure Function App subscribes to the Event Grid topic and acts as the intelligent trigger layer. It evaluates the drift signal's severity, applies any business rules (e.g. minimum data volume for retraining to make sense), and if conditions are met, triggers the retraining pipeline via its Batch Endpoint. The entire cycle — from drift detection to a new training job being submitted — requires zero human involvement. This is a genuinely event-driven retraining loop, not a scheduled job.

05
deliberate human gate
Human-Validated Promotion — Governance by Design

After retraining completes, a custom validation step gates promotion to production. A human reviewer evaluates model outputs, validation metrics, and business-logic correctness before the new model version is promoted to the production endpoint. This gate is a deliberate architectural decision, not a shortcoming. Automated metrics can confirm a model retrained successfully, but they cannot confirm it is appropriate for production in the context of recent business, data, or regulatory changes. The platform is fully capable of automating this gate — the infrastructure exists — and that is the planned Phase 2 enhancement once acceptance criteria are formally encoded.

Architectural Decision — Human Promotion Gate
Why human validation is the right choice at this stage

Fully automated model promotion is achievable on this platform — the ML Registry, endpoint deployment pipeline, and approval infrastructure are all in place to support it. The decision to require human review at promotion reflects engineering maturity: automated metrics confirm that a model trained correctly, but a human reviewer can assess whether the model should go live in the context of current business conditions, recent data changes, and regulatory posture. When formal acceptance criteria are encoded and validated over multiple retraining cycles, full automation of the promotion gate is a straightforward Phase 2 implementation on existing infrastructure.

✓
Best Practice — Confirmed

The Event Grid + Function App trigger pattern decouples drift detection from retraining execution. This is architecturally cleaner than a scheduled polling approach — retraining runs exactly when needed, not on a fixed schedule regardless of drift state. It also means the trigger logic can evolve independently of the ML pipeline code.

05 — Challenges & Roadmap

Known Challenges
& What Comes Next

A production platform surfaces real engineering challenges. We document them transparently here — with clear mitigations and a Phase 2 roadmap that builds directly on the infrastructure already in place.

ChallengeImpactMitigationSeverity
Private DNS Split-Brain
Multi-region Private Endpoint DNS can conflict if spokes register records independently.		 New Private Endpoints fail to resolve unless hub DNS Zone links are configured correctly. Terraform outputs from the bootstrap layer pipe hub DNS Zone IDs directly into the AI platform module. All Private Endpoint DNS registration targets hub zones only — enforced by Checkov policy gate in CI. ● High
Azure OpenAI TPM Limits
Tokens-per-minute and requests-per-minute are set at the deployment level and do not auto-scale.		 Request failures under load spikes if quota is not correctly pre-provisioned per deployment. AOAI quotas are managed via Terraform — changes require a reviewed PR. PTU (Provisioned Throughput Units) are planned for high-volume workloads. APIM rate limiting policies are Phase 2 configuration on the already-provisioned gateway. ● High
ML Compute Cold Start
Scale-to-zero compute clusters take 3–8 minutes to provision for the first job.		 Poor developer experience for interactive or frequent training jobs. Dev clusters maintain a minimum of 1 warm node during business hours. Production inference uses Managed Online Endpoints (always-on) — not cluster compute. This separation is a deliberate design choice in the custom Terraform configuration. ● Medium
AVM Module Upgrade Lag
AVM major version updates may include breaking changes requiring Terraform refactoring.		 Security-critical AVM updates may be delayed if breaking changes require significant effort. Dependabot monitors AVM module versions weekly. Patch and minor upgrades are applied automatically via PR. Major version upgrades are planned as sprint items with a dedicated testing window in non-prod before production promotion. ● Medium
Model Promotion Acceptance Criteria
Human validation gate requires reviewers to make judgement calls without formally encoded criteria.		 Inconsistent promotion decisions across reviewers; gate may slow down routine retraining cycles. The current gate correctly prioritises safety over speed. Phase 2 work is to formalise acceptance criteria as measurable thresholds — enabling automation of the gate for routine retraining while preserving human review for out-of-distribution model behaviour. ● Medium
Terraform State Drift
Azure auto-healing or manual portal changes can cause IaC state drift without triggering a pipeline.		 Undetected drift causes apply failures or silent configuration inconsistencies on next run. Azure Policy deny blocks manual resource creation in production. Azure Locks protect state storage and Key Vault. Scheduled drift detection (GitHub Actions cron running terraform plan every 6 hours) is a Phase 2 operational enhancement. ● Low

The current human validation gate is replaced with an encoded acceptance criteria check once thresholds are formally defined and validated across multiple retraining cycles. The deployment pipeline already supports automated promotion — this is a policy change, not an engineering build.
Phase 2 · Operations
Scheduled Drift Detection
GitHub Actions cron running terraform plan every 6 hours across all environments. Non-empty plans alert to Teams with a diff summary. Enables proactive detection of configuration drift before it causes pipeline failures.
